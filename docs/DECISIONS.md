# Implementation Decisions

- Admission fees are deducted inside `Store.submit_claim`. This satisfies the requirement that the fee is charged at submission regardless of admission outcome and keeps rejected claims accountable even before the coordinator polls.
- Challenge graph changes are append-only citation overrides in `claim_cite_overrides`, not mutations to admitted `claims`. Settlement snapshots apply the latest override so the challenge graph is post-resolution while the base claim ledger stays immutable.
- Payouts are transferred from pseudo escrow accounts such as `ESCROW:<task_id>`. The escrow debit from the sponsor is recorded at task creation, but pseudo accounts are not identities because the spec only requires identity balances.
- The synthetic fact task uses independent fact subtasks plus an answer subtask depending on all facts. This exercises leases and the dependency DAG while allowing multiple local workers to race on discovery.
- The optional LLM path is implemented behind the verifier interface with an OpenAI-compatible HTTP hook. Tests skip it when `VSCP_LLM` is unset and use a crafted completion function when enabled.
- `FAIL` claims are verified negative results: `MockVerifier` admits one only if the body is a decoy per the answer key *and* appears in the evidence spans. Bodies that merely claim failure are rejected. Scripted workers publish a `FAIL` (with an evidence ref, no subtask) for every decoy hit and read others' FAILs through the metered `fetch` to prune their candidate sets — this is what makes the steering pool (γ) flow end-to-end instead of burning.
- Cited admitted claims count as support spans at admission. Consequently `ANSWER` claims carry provenance via `cites` only — no re-attached upstream evidence, and a compact O(1) body verified by coverage (every key fact present in the spans) instead of verbatim listing. The spec's verbatim-listing rule (§3) conflicts with the 600-char body cap (§6.1) for K ≥ 8; coverage via cites resolves the conflict and removes the 8-ref cap on K entirely.
- Cites must target claims of the same task (`cross_task_cite` rejection); the settlement snapshot is per-task, so cross-task edges would dangle.
- `resolve_ref_span` returns the *minimal* valid head→tail span (deterministic tie-break on earliest head). Returning the first match let an author pick a common head and a distant tail to capture text they never localized.
- `store.fetch` serves ADMITTED claim bodies only; PENDING/REJECTED bodies would leak pre-admission information. Evidence documents are always servable.
- Pseudo accounts (`ESCROW:*`, `BOND:*`, `FEE_POOL`) are tracked in a `pseudo_accounts` table with a non-negative CHECK; admission fees are routed into `FEE_POOL` so refunds draw against a real balance. A global conservation test asserts identities + pseudo accounts + burns equal total minted.
- Challenges resolve atomically: verdict is computed read-only, then bond, cite override, payouts, and the challenge row commit in a single transaction. An upheld slash is capped at the target author's balance rather than wedging resolution. Bond asymmetry is deliberate: a failed under-citation bond burns (the check is deterministic; frivolous filings are spam), while a failed materiality bond pays the target author (the challenger consumed verifier work against a real citation).
- The scripted hoarder is a *racing* hoarder: it free-rides on published facts via `fetch` and submits an unleased `ANSWER` the moment all facts are admitted, paying only the admission fee. It loses the race to the honest finisher and forfeits the fee, which is the incentive claim the demo actually needs to exercise.

## Phase 2

- `settle(snapshot, version=None)`: an explicit `version` argument takes precedence over the snapshot's `settlement_version` key (caller intent wins, e.g. for v1-vs-v2 comparisons on the same snapshot); absent both, the default is `SETTLEMENT_VERSION = 2`. The coordinator stamps `tasks.settlement_version` at close and `store.snapshot` carries it.
- Steering v2 ties endorsement to *informed work*: a fetch only counts if the reader later (epoch-wise: fetch epoch <= productive claim's `epoch_submitted`) authored a positively-flowing claim whose docs overlap the FAIL's docs, and each productive reader's endorsement is normalized to `1_000_000` units. In the seed-42 demo the finisher is effectively the only eligible reader (per-doc FACT authors author the FAILs on their own doc, which never count), so gamma flows through the finisher's normalized endowment - still fully paid, no longer collusion-priced.
- `delm_market_sim` was not available in this environment, so `tools/sim/sim.py` is an *independently written* float-arithmetic reference implementation of the same mechanism (recursive flow propagation, per-author accumulation) rather than a vendored copy. The equivalence harness's purpose - two independent implementations cross-checking each other - is preserved; if the original sim surfaces, drop it in and rerun the same scenarios.
- Scenario claims without an explicit `"epoch"` key use their list index as `epoch_submitted` (submission order doubles as the epoch).
- Challenge slashes are capped at the target author's balance inside the single resolution transaction (Phase 1 decision, restated here because §2 of the Phase 2 spec depends on resolved challenge graphs).
- ANSWER-body discipline: admission requires `^ANSWER: ` and the existing 600-char cap. Semantic verification of answer bodies against cited claims is performed by the hardened LLM verifier (§5); the mock path performs the coverage check only (every key fact present in the support spans) and deliberately nothing more.
- The corpus documents are original texts written for this repo (effectively CC0) of ~1.0-1.3 kB each, slightly below the spec's 2-8 kB suggestion, keeping fake-LLM CI prompts compact; the loader imposes no size constraint, so larger public-domain texts drop in unchanged.
- `legion eval` selects its backend automatically: the real OpenAI-compatible endpoint iff `VSCP_LLM=1`, otherwise a deterministic fake `complete` stub that answers from the fixture's gold facts. The stub is also what CI runs.
- The LLM worker's local verbatim check runs before `submit_claim`, so hallucinated sentences cost zero fees; after 3 failed extraction attempts (initial + two resubmissions) the subtask is abandoned for that worker. The per-worker call budget (`VSCP_MAX_LLM_CALLS_PER_WORKER`, default 30) is shared across extraction and synthesis calls; verifier calls are counted separately by the eval harness.
- A realdoc `FAIL` is a *verified irrelevance statement* (a sentence asserted irrelevant despite appearing relevant), exposed via `LLMWorker.submit_irrelevance_fail` and admitted only if the verifier supports it; model output that merely fails the verbatim check is a worker error and is never published.
